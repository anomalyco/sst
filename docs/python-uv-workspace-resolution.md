# Python UV Workspace Resolution Architecture

This document describes how SST resolves Python UV workspaces and handles workspace dependencies for Lambda deployment.

## Overview

SST supports UV workspaces for Python monorepo projects. When building a Lambda function, SST:

1. Locates the handler file and its `pyproject.toml`
2. Finds the UV workspace root (if any)
3. Exports dependencies via `uv export`
4. Filters out workspace package editable installs (which don't work in Lambda)
5. Copies workspace package source code into the artifact
6. Installs external dependencies via `uv pip install`

## Directory Structure

A typical UV workspace monorepo:

```
monorepo/
‚îú‚îÄ‚îÄ pyproject.toml              # Root workspace config
‚îú‚îÄ‚îÄ uv.lock                     # Workspace-wide lockfile
‚îú‚îÄ‚îÄ services/
‚îÇ   ‚îú‚îÄ‚îÄ api/
‚îÇ   ‚îÇ   ‚îú‚îÄ‚îÄ pyproject.toml      # Service-specific dependencies
‚îÇ   ‚îÇ   ‚îî‚îÄ‚îÄ handler.py
‚îÇ   ‚îî‚îÄ‚îÄ auth/
‚îÇ       ‚îú‚îÄ‚îÄ pyproject.toml
‚îÇ       ‚îî‚îÄ‚îÄ handler.py
‚îî‚îÄ‚îÄ shared/                     # Shared library package
    ‚îú‚îÄ‚îÄ pyproject.toml
    ‚îú‚îÄ‚îÄ __init__.py
    ‚îî‚îÄ‚îÄ utils.py
```

### Root pyproject.toml

The root `pyproject.toml` defines which directories are workspace members:

```toml
[tool.uv.workspace]
members = ["services/*", "shared"]
```

### Service pyproject.toml

Each service declares its dependencies and references workspace packages using `{ workspace = true }`:

```toml
[project]
name = "api-service"
dependencies = [
    "fastapi>=0.104.0",
    "shared",  # Reference to workspace package
]

[tool.uv.sources]
shared = { workspace = true }
```

The `{ workspace = true }` syntax tells UV to resolve `shared` from the workspace members rather than PyPI.

## Build Process

### Step 1: Handler Resolution

When SST receives a handler like `services/api/handler.main`:

1. Locates `services/api/handler.py`
2. Walks **up** the directory tree to find the nearest `pyproject.toml`
3. Result: `services/api/pyproject.toml`

### Step 2: Workspace Root Detection

SST walks up from the service's `pyproject.toml` looking for a parent `pyproject.toml` that defines `[tool.uv.workspace]` members.

**Important:** SST stops at the **first** UV workspace it finds, not the topmost. This matters when your project is nested inside another UV workspace (e.g., examples inside a larger repo).

```
/some/outer/workspace/          # Has [tool.uv.workspace] - IGNORED
  ‚îî‚îÄ‚îÄ your-project/
      ‚îî‚îÄ‚îÄ monorepo/             # Has [tool.uv.workspace] - USED ‚úì
          ‚îî‚îÄ‚îÄ services/
              ‚îî‚îÄ‚îÄ api/
                  ‚îî‚îÄ‚îÄ pyproject.toml  ‚Üê Starting point
```

### Step 3: Dependency Export

SST runs `uv export` from the service directory to generate a `requirements.txt`. UV includes all dependencies, including workspace packages as editable installs:

```text
# Generated by uv export
fastapi==0.104.1
pydantic==2.5.0
-e file:///path/to/monorepo/services/api
-e file:///path/to/monorepo/shared
```

### Step 4: Filtering Workspace Packages

Editable installs (`-e`) reference local file paths that don't exist in Lambda. SST:

1. **Removes** all `-e` lines and local path references
2. **Records** those paths for source copying
3. **Writes** a filtered `requirements.txt` with only external dependencies

Filtered result:
```text
fastapi==0.104.1
pydantic==2.5.0
```

### Step 5: Copying Workspace Package Sources

For each workspace package, SST copies the Python source code into the Lambda artifact. SST detects the package's source layout automatically.

#### Supported Source Layouts

**1. src-layout (PEP 517 recommended)**
```
shared/
‚îú‚îÄ‚îÄ pyproject.toml
‚îî‚îÄ‚îÄ src/
    ‚îî‚îÄ‚îÄ shared/
        ‚îú‚îÄ‚îÄ __init__.py
        ‚îî‚îÄ‚îÄ utils.py
```
‚Üí Copies `src/shared/` to artifact as `shared/`

**2. Flat src-layout**
```
shared/
‚îú‚îÄ‚îÄ pyproject.toml
‚îî‚îÄ‚îÄ src/
    ‚îú‚îÄ‚îÄ __init__.py
    ‚îî‚îÄ‚îÄ utils.py
```
‚Üí Copies `src/` to artifact as `shared/` (using package name from pyproject.toml)

**3. Package directory at root**
```
shared-pkg/
‚îú‚îÄ‚îÄ pyproject.toml
‚îî‚îÄ‚îÄ shared/
    ‚îú‚îÄ‚îÄ __init__.py
    ‚îî‚îÄ‚îÄ utils.py
```
‚Üí Copies `shared/` to artifact as `shared/`

**4. Flat package directory**
```
shared/
‚îú‚îÄ‚îÄ pyproject.toml
‚îú‚îÄ‚îÄ __init__.py
‚îî‚îÄ‚îÄ utils.py
```
‚Üí Copies entire `shared/` to artifact as `shared/`

### Step 6: Installing External Dependencies

Finally, SST installs the filtered dependencies:

```bash
uv pip install \
    -r requirements-filtered.txt \
    --target /path/to/artifact \
    --python-platform x86_64-unknown-linux-gnu
```

The `--python-platform` flag ensures Lambda-compatible (Linux) wheels are installed.

After the build, the Lambda artifact contains:

Your handler can now `from shared.utils import ...` and it works in Lambda.

## Per-Package Dependency Isolation

The key benefit of this architecture: **each Lambda only includes its own dependencies**.

| Service | Dependencies | Bundle Size |
|---------|-------------|-------------|
| api | fastapi, boto3, shared | ~19 MB |
| auth | pyjwt, bcrypt, shared | ~337 KB |
| worker | celery, redis, shared | ~2.8 MB |

Without isolation, all three would be ~22 MB each!

## Troubleshooting

### "No module named 'shared'"

The workspace package source wasn't copied. Check:

1. **Package has pyproject.toml** - SST needs this to determine the package name
2. **Source layout is supported** - See "Supported Source Layouts" above
3. **Check SST logs** - Look for "could not determine source layout for workspace package"

### Wrong workspace root detected

SST found the wrong `[tool.uv.workspace]`. Check:

1. **Your workspace has members defined** - The root `pyproject.toml` must have `[tool.uv.workspace]` with `members = [...]`
2. **No outer workspace interference** - If nested in another UV workspace, ensure your workspace is the first one found walking up

### Editable install errors in Lambda

The filtering didn't remove all local paths. Check:

1. **Lambda logs** - Look for errors about `-e` or `file://` paths
2. **Generated requirements.txt** - Should only contain package names and versions, no paths
