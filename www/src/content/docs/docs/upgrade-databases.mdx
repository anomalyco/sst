---
title: Upgrade Databases
description: What to expect when upgrading your database engine version, instance type, or other settings.
---

When you change the `version`, `instance`, or other settings of your database component, SST applies the update immediately on the next `sst deploy`. This applies to all database components: [`Postgres`](/docs/component/aws/postgres/), [`Mysql`](/docs/component/aws/mysql/), and [`Aurora`](/docs/component/aws/aurora/).

This page covers what happens during these upgrades and how to minimize downtime.

---

## What happens during an upgrade

When you change a prop like `version` or `instance` and run `sst deploy`, the database instance is modified in-place. During this time:

- The database **restarts**, so there will be some downtime.
- Depending on the change, this can take anywhere from a few minutes to **30 minutes or more**.
- Your application will get connection errors until the database is back up.

This is the default behavior for all database components.

```ts title="sst.config.ts" {3}
const database = new sst.aws.Postgres("MyDatabase", {
  vpc,
  version: "17",
});
```

For dev and staging environments, this is usually fine.

---

## Reducing downtime

There are a couple of strategies to reduce or avoid downtime during upgrades.

---

### Multi-AZ

[`Postgres`](/docs/component/aws/postgres/) and [`Mysql`](/docs/component/aws/mysql/) support [Multi-AZ deployments](https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/Concepts.MultiAZ.html). When enabled, AWS maintains a standby replica in a different availability zone. During upgrades, AWS can failover to the standby to reduce downtime.

```ts title="sst.config.ts" {4}
const database = new sst.aws.Postgres("MyDatabase", {
  vpc,
  version: "17",
  multiAz: true,
});
```

Multi-AZ roughly doubles the cost of the database since two instances are running at all times. But it also provides automatic failover if the primary instance fails, so it's a good idea for production databases regardless of upgrades.

---

### Blue/Green deployments

[`Postgres`](/docs/component/aws/postgres/) and [`Mysql`](/docs/component/aws/mysql/) support [AWS RDS Blue/Green deployments](https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/blue-green-deployments.html) through the `upgrade` prop.

```ts title="sst.config.ts" {4}
const database = new sst.aws.Postgres("MyDatabase", {
  vpc,
  version: "17",
  upgrade: "blue-green",
});
```

This works the same way for [`Mysql`](/docs/component/aws/mysql/).

```ts title="sst.config.ts" {4}
const database = new sst.aws.Mysql("MyDatabase", {
  vpc,
  version: "8.4.4",
  upgrade: "blue-green",
});
```

When you deploy with `upgrade: "blue-green"`, the following happens:

1. A **staging (green)** copy of your database is created from the current **production (blue)** instance.
2. The requested changes are applied to the green instance.
3. AWS runs [guardrail checks](https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/blue-green-deployments-switching.html#blue-green-deployments-switching-guardrails) to verify the green instance is healthy.
4. The green instance is **promoted** to become the production instance.
5. The old blue instance and the deployment are cleaned up.

Your database endpoint stays the same throughout this process, so no application changes are needed.

---

#### Trade-offs

Blue/Green deployments minimize downtime but come with some trade-offs.

- **Longer deploy times.** Expect 20-30 minutes because of the staging instance creation and switchover process.
- **No read replicas.** Blue/Green deployments are not compatible with databases that have read replicas. AWS will reject the deployment.
- **No custom health checks.** The switchover happens automatically after AWS's guardrail checks pass. You won't be able to run your own checks on the green instance before it's promoted.
- **Manual cleanup on failure.** If the deployment fails midway, the green instance and the deployment may need to be cleaned up manually in the AWS console.
- **Resource ID may change.** The RDS resource ID may change after the switchover. The endpoint and connection details are preserved by AWS.
- **Instance-only.** Blue/Green deployments only work with standalone RDS instances ([`Postgres`](/docs/component/aws/postgres/) and [`Mysql`](/docs/component/aws/mysql/)). They are not supported for [`Aurora`](/docs/component/aws/aurora/) clusters due to a [Pulumi provider limitation](https://github.com/pulumi/pulumi-aws/issues/6145).

## Upgrading step by step

Here's the recommended workflow for upgrading a production database.

---

### 1. Check compatibility

Make sure the upgrade path is supported. Not all version jumps are allowed — for example, you might need to go through intermediate versions.

- [PostgreSQL upgrade paths](https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/USER_UpgradeDBInstance.PostgreSQL.html)
- [MySQL upgrade paths](https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/USER_UpgradeDBInstance.MySQL.html)
- [Aurora PostgreSQL upgrade paths](https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/USER_UpgradeDBInstance.PostgreSQL.html)
- [Aurora MySQL upgrade paths](https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/AuroraMySQL.Updates.MajorVersionUpgrade.html)

---

### 2. Test in a non-production stage

Deploy the version change to a dev or staging stage first to make sure everything works.

```ts title="sst.config.ts"
const database = new sst.aws.Postgres("MyDatabase", {
  vpc,
  version: "17",
});
```

---

### 3. Deploy to production

For production, consider using Blue/Green deployments to minimize downtime.

```ts title="sst.config.ts" {4}
const database = new sst.aws.Postgres("MyDatabase", {
  vpc,
  version: "17",
  upgrade: "blue-green",
});
```

:::tip
You can set `upgrade` conditionally so dev environments use in-place upgrades while production uses Blue/Green.

```ts
upgrade: $app.stage === "production" ? "blue-green" : "in-place",
```
:::

---

### 4. Clean up

After the upgrade is complete, you can remove the `upgrade` prop. It only affects how future changes are applied.
