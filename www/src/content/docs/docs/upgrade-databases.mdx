---
title: Upgrade Databases
description: What to expect when upgrading your database engine version or instance type, and how to minimize downtime.
---

When you change the `version` or `instance` of your database component, SST applies the update immediately on the next `sst deploy`. This applies to [`Postgres`](/docs/component/aws/postgres/), [`Mysql`](/docs/component/aws/mysql/), and [`Aurora`](/docs/component/aws/aurora/).

:::note
Other settings like `storage` or `password` do not cause downtime.
:::

This page covers what to expect during these upgrades and the different strategies for handling them.

---

## In-place upgrades

By default, when you change the `version` or `instance` prop and deploy, the update is applied directly to the running instance. The database restarts while the update is being applied, so your application will lose connectivity for a period of time. Depending on the change, this can take anywhere from a few minutes up to 30 minutes or more.

```ts title="sst.config.ts" {3}
const database = new sst.aws.Postgres("MyDatabase", {
  vpc,
  version: "17",
});
```

This is the default for all database components: [`Postgres`](/docs/component/aws/postgres/), [`Mysql`](/docs/component/aws/mysql/), and [`Aurora`](/docs/component/aws/aurora/). If you're upgrading a production database, read on for strategies to reduce downtime.

---

## Reducing downtime

### Multi-AZ

[`Postgres`](/docs/component/aws/postgres/) and [`Mysql`](/docs/component/aws/mysql/) support [Multi-AZ deployments](https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/Concepts.MultiAZ.html). When enabled, AWS maintains a standby replica in a different availability zone. During an upgrade, AWS applies the changes to the standby first, then fails over to it. This reduces downtime to around **60 seconds** — the time it takes for the failover — instead of the full duration of the upgrade.

```ts title="sst.config.ts" {4}
const database = new sst.aws.Postgres("MyDatabase", {
  vpc,
  version: "17",
  multiAz: true,
});
```

Multi-AZ roughly doubles the cost of the database since two instances are running at all times. But it also provides automatic failover if the primary instance fails, making it a good idea for production databases regardless of upgrades.

If you don't want to keep Multi-AZ enabled permanently, you can enable it temporarily for the upgrade. Deploy with `multiAz: true` first and wait for the standby to be created and synced. Then change the `version` and deploy again. After the upgrade completes, you can set `multiAz` back to `false`.

:::note
[`Aurora`](/docs/component/aws/aurora/) does not currently support the `multiAz` prop, but Aurora clusters inherently store data across multiple AZs. Adding replicas through the `replicas` prop provides similar failover behavior during upgrades. If you'd like first-class `multiAz` support for Aurora, [let us know](https://github.com/sst/sst/issues/0000).
:::

### Blue/Green deployments

[`Postgres`](/docs/component/aws/postgres/) and [`Mysql`](/docs/component/aws/mysql/) support [AWS RDS Blue/Green deployments](https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/blue-green-deployments.html) through the `upgrade` prop. This creates a complete staging copy of your database, applies the changes there, and then switches over with minimal downtime.

```ts title="sst.config.ts" {4}
const database = new sst.aws.Postgres("MyDatabase", {
  vpc,
  version: "17",
  upgrade: "blue-green",
});
```

This works the same way for [`Mysql`](/docs/component/aws/mysql/).

```ts title="sst.config.ts" {4}
const database = new sst.aws.Mysql("MyDatabase", {
  vpc,
  version: "8.4.4",
  upgrade: "blue-green",
});
```

When you deploy with `upgrade: "blue-green"`, the following happens:

1. A **staging (green)** copy of your database is created from the current **production (blue)** instance.
2. The requested changes are applied to the green instance.
3. AWS runs [guardrail checks](https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/blue-green-deployments-switching.html#blue-green-deployments-switching-guardrails) to verify the green instance is healthy.
4. The green instance is **promoted** to become the production instance.
5. The old blue instance and the deployment are cleaned up.

Your database endpoint stays the same throughout this process, so no application changes are needed.

There are some things to keep in mind when using Blue/Green deployments:

- **Longer deploy times.** Expect 20-30 minutes because of the staging instance creation and switchover process. The actual downtime during the switchover is brief, but the overall deploy takes significantly longer than in-place.
- **No read replicas.** Blue/Green deployments are not compatible with databases that have read replicas. AWS will reject the deployment.
- **No custom health checks.** The switchover happens automatically after AWS's guardrail checks pass. You can't run your own checks on the green instance before it's promoted.
- **Manual cleanup on failure.** If the deployment fails midway, the green instance and the deployment may need to be cleaned up manually in the AWS console.
- **Resource ID may change.** The RDS resource ID may change after the switchover. The endpoint and connection details are preserved by AWS.
- **Instance-only.** Blue/Green deployments only work with [`Postgres`](/docs/component/aws/postgres/) and [`Mysql`](/docs/component/aws/mysql/). They are not supported for [`Aurora`](/docs/component/aws/aurora/) clusters due to a [Pulumi provider limitation](https://github.com/pulumi/pulumi-aws/issues/6145).

## Upgrading step by step

Here's the recommended workflow for upgrading a production database.

---

### 1. Check compatibility

Make sure the upgrade path is supported. Not all version jumps are allowed — you might need to go through intermediate versions.

- [PostgreSQL upgrade paths](https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/USER_UpgradeDBInstance.PostgreSQL.html)
- [MySQL upgrade paths](https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/USER_UpgradeDBInstance.MySQL.html)
- [Aurora PostgreSQL upgrade paths](https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/USER_UpgradeDBInstance.PostgreSQL.html)
- [Aurora MySQL upgrade paths](https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/AuroraMySQL.Updates.MajorVersionUpgrade.html)

---

### 2. Pick a strategy

Decide how you want to handle the upgrade based on your downtime tolerance.

- **In-place** — simplest, but expect downtime during the upgrade. Good for dev and staging.
- **Multi-AZ** — reduces downtime to about 60 seconds by failing over to an upgraded standby. Can be enabled temporarily for the upgrade.
- **Blue/Green** — minimizes downtime by creating a staging copy and switching over. Takes longer overall (20-30 min) but the application sees very little interruption. Only available for `Postgres` and `Mysql`.

---

### 3. Test in a non-production stage

Deploy the version change to a dev or staging stage first to make sure everything works.

```ts title="sst.config.ts"
const database = new sst.aws.Postgres("MyDatabase", {
  vpc,
  version: "17",
});
```

---

### 4. Deploy to production

Apply your chosen strategy to the production deploy.

```ts title="sst.config.ts" {4}
const database = new sst.aws.Postgres("MyDatabase", {
  vpc,
  version: "17",
  upgrade: "blue-green",
});
```

:::tip
You can set `upgrade` conditionally so dev environments use in-place upgrades while production uses Blue/Green.

```ts
upgrade: $app.stage === "production" ? "blue-green" : "in-place",
```
:::
