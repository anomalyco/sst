title: Upgrade Databases
description: What to expect when upgrading your database engine version or instance type, and how to minimize downtime.
---

When you change the `version`, `instance` or `engine` of your database component, SST applies the update immediately on the next `sst deploy`. This applies to [`Postgres`](/docs/component/aws/postgres/), [`Mysql`](/docs/component/aws/mysql/), [`Aurora`](/docs/component/aws/aurora/), [`Redis`](/docs/component/aws/redis/), and [`OpenSearch`](/docs/component/aws/open-search/).

:::note
Not all settings cause downtime. Check the component docs for which fields trigger a restart.
:::

This page covers what to expect during these upgrades and the different strategies for handling them.

---

## In-place upgrades

By default, when you change a property that requires restart, the update is applied immediately to the running instance when running `sst deploy`. The database restarts while the update is being applied, so your application will lose connectivity for a period of time. Depending on the change, this can take anywhere up to 30 minutes.

```ts title="sst.config.ts" {3}
const database = new sst.aws.Postgres("MyDatabase", {
  vpc,
  version: "17",
});
```

This is the default for [`Postgres`](/docs/component/aws/postgres/), [`Mysql`](/docs/component/aws/mysql/), [`Aurora`](/docs/component/aws/aurora/), [`Redis`](/docs/component/aws/redis/), and [`OpenSearch`](/docs/component/aws/open-search/).

---

## Reducing downtime

If you're upgrading a production database there are a couple of strategies to reduce downtime.

### Multi-AZ

[`Postgres`](/docs/component/aws/postgres/), [`Mysql`](/docs/component/aws/mysql/), and [`Redis`](/docs/component/aws/redis/) support Multi-AZ deployments. When enabled, AWS maintains a standby replica in a different availability zone. During the upgrade, AWS applies the changes to the standby first, then fails over to it. This reduces downtime to around **60-120 seconds** (the time it takes for the failover) instead of the full duration of the upgrade.

```ts title="sst.config.ts" {4}
const database = new sst.aws.Postgres("MyDatabase", {
  vpc,
  version: "17",
  multiAz: true,
});
```

:::caution
Upgrading a major database version will restart both instances, making Multi-AZ useless in this situation.
:::

Multi-AZ roughly doubles the cost since a standby instance is running at all times. But it also provides automatic failover if the primary fails, making it a good idea for production workloads regardless of upgrades.

If you don't want to keep Multi-AZ enabled permanently, you can enable it temporarily for the upgrade. Deploy with `multiAz: true` first and wait for the standby to be created and synced. Then make the change and deploy again. After the upgrade completes, you can set `multiAz` back to `false`.

:::note
Aurora does not currently support the `multiAz` prop, but Aurora clusters inherently store data across multiple AZs. Adding replicas through the `replicas` prop provides similar failover behavior during upgrades.
:::

### Blue/Green deployments

[`Postgres`](/docs/component/aws/postgres/) and [`Mysql`](/docs/component/aws/mysql/) support [AWS RDS Blue/Green deployments](https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/blue-green-deployments.html) through the `blueGreen` prop. This creates a complete staging copy of your database, applies the changes there, and then switches over with minimal downtime.

```ts title="sst.config.ts" {4}
const database = new sst.aws.Postgres("MyDatabase", {
  vpc,
  version: "17",
  blueGreen: true,
});
```

This works the same way for [`Mysql`](/docs/component/aws/mysql/).

```ts title="sst.config.ts" {4}
const database = new sst.aws.Mysql("MyDatabase", {
  vpc,
  version: "8.4.4",
  blueGreen: true,
});
```

When you deploy with `blueGreen: true`, the following happens:

1. A **staging (green)** copy of your database is created from the current **production (blue)** instance.
2. The requested changes are applied to the green instance.
3. AWS runs [guardrail checks](https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/blue-green-deployments-switching.html#blue-green-deployments-switching-guardrails) to verify the green instance is healthy.
4. The green instance is **promoted** to become the production instance.
5. The old blue instance and the deployment are cleaned up.

Your database endpoint stays the same throughout this process, so no application changes are needed.

When using Blue/Green deployments expect longer deploy times because of the staging instance creation and switchover process. The actual downtime during the near zero, but the overall deploy takes significantly longer than in-place.

Also, Blue/Green deployments are not compatible with databases that have read replicas. AWS will reject the deployment.

Read more about AWS RDS Blue/Green deployments in [this Pulumi article](https://www.pulumi.com/blog/aws-rds-blue-green-deployment-updates).

## Upgrading step by step

Here's the recommended workflow for upgrading a production database.

### 1. Check compatibility

Make sure the upgrade path is supported. Not all version jumps are allowed — you might need to go through intermediate versions.

- [PostgreSQL upgrade paths](https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/USER_UpgradeDBInstance.PostgreSQL.html)
- [MySQL upgrade paths](https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/USER_UpgradeDBInstance.MySQL.html)
- [Aurora PostgreSQL upgrade paths](https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/USER_UpgradeDBInstance.PostgreSQL.html)
- [Aurora MySQL upgrade paths](https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/AuroraMySQL.Updates.MajorVersionUpgrade.html)

### 2. Pick a strategy

Decide how you want to handle the upgrade based on your downtime tolerance.

- **In-place** — simplest, but plan for downtime during the upgrade.
- **Multi-AZ** — reduces downtime to about 60 seconds by failing over to an upgraded standby. Can be enabled temporarily for the upgrade.
- **Blue/Green** — minimizes downtime by creating a staging copy and switching over. The overall deploy takes longer (20-30 min) but the application sees almost zero interruption.

| Component | In-place | Multi-AZ | Blue/Green |
| --- | --- | --- | --- |
| [`Postgres`](/docs/component/aws/postgres/) | ✓ | ✓ | ✓ |
| [`Mysql`](/docs/component/aws/mysql/) | ✓ | ✓ | ✓ |
| [`Aurora`](/docs/component/aws/aurora/) | ✓ | ✕ | ✕ |
| [`Redis`](/docs/component/aws/redis/) | ✓ | ✓ | ✕ |
| [`OpenSearch`](/docs/component/aws/open-search/) | ✓ | ✕ | ✕ |

### 3. Test in a non-production stage

Deploy the version change to a dev or staging stage first to make sure everything works.

```diff lang="ts" title="sst.config.ts"
  const database = new sst.aws.Postgres("MyDatabase", {
    vpc,
-   version: "16",
+   version: "17",
    blueGreen: true,
  });
```

### 4. Deploy to production

Apply your chosen strategy to the production deploy.

```ts title="sst.config.ts" {4}
const database = new sst.aws.Postgres("MyDatabase", {
  vpc,
  version: "17",
  blueGreen: true,
});
```

You can set `blueGreen` conditionally so only production uses Blue/Green deployments.

```ts
blueGreen: $app.stage === "production",
```
