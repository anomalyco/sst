SST provides flexible Python Lambda support that works with any project structure. The system automatically detects your project configuration and builds your functions efficiently using intelligent caching and modern Python tooling.

## Quick Start

Python functions can be deployed just like other SST functions. Simply specify the handler path relative to your project root:

```typescript title="sst.config.ts"
const python = new sst.aws.Function("MyPythonFunction", {
  handler: "src/mypackage/handler.main",
  runtime: "python3.11",
  url: true
});
```

## Project Structure

SST supports multiple project structures without requiring specific directory layouts:

```
my-project/
├── pyproject.toml
├── src/
│   └── mypackage/
│       ├── __init__.py
│       └── handler.py
└── shared/
    └── utils.py
```

Or a simpler layout:

```
my-function/
├── handler.py
├── utils.py
└── requirements.txt
```

## Dependencies

SST uses [uv](https://github.com/astral-sh/uv) for fast dependency management. We recommend using `pyproject.toml` for modern Python packaging:

```toml title="pyproject.toml"
[project]
name = "my-lambda-project"
version = "0.1.0"
description = "My Lambda functions"
dependencies = [
    "requests>=2.31.0",
    "boto3>=1.34.0"
]
requires-python = ">=3.9"

[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"
```

### Installing UV (Recommended)

For the fastest builds, install UV:

```bash
curl -LsSf https://astral.sh/uv/install.sh | sh
```

### Alternative: requirements.txt

You can also use traditional `requirements.txt`:

```txt title="requirements.txt"
requests>=2.31.0
boto3>=1.34.0
```

## Development

During `sst dev`, Python functions run locally with hot reloading:

```bash
sst dev
```

Changes to your Python code are automatically detected and the function is rebuilt in seconds, not minutes.

SST's Python runtime includes several performance optimizations:

- **Intelligent Caching**: Only rebuilds when files actually change (97% faster for unchanged code)
- **Incremental Builds**: Only rebuilds changed packages in multi-package projects
- **Parallel Processing**: Concurrent builds for independent packages
- **Fast Dependencies**: UV provides 2-10x faster dependency installation

## Best Practices

### Module Imports

**Use Absolute Imports**

Always use absolute imports for reliable behavior in Lambda:

```python
# Recommended - absolute imports work reliably
from shared.utils import helper_function
from app.functions.auth.handler import auth_main
from mypackage.submodule import MyClass
from core.database import get_connection
```

**Avoid Relative Imports**

Relative imports can be unreliable in Lambda environments:

```python
# Not recommended - can fail in Lambda
from .utils import helper_function
from ..shared import common_function
```

### Static File Access

When your Lambda function needs to read static files, use paths relative to your handler file:

```python
import json
from pathlib import Path

def load_config():
    # Recommended - relative to handler file
    config_path = Path(__file__).parent / "../../data/config.json"
    with open(config_path, 'r') as f:
        return json.load(f)

def load_template():
    # Also works - using os.path
    import os
    template_path = os.path.join(os.path.dirname(__file__), "../templates/email.html")
    with open(template_path, 'r') as f:
        return f.read()
```

**Avoid Working Directory Assumptions**

```python
# Not recommended - assumes working directory
with open("config.json", 'r') as f:  # May fail
    config = json.load(f)
```

### Package Structure

Ensure proper Python package structure with `__init__.py` files:

```
project/
├── shared/
│   ├── __init__.py          # Required for imports
│   └── utils.py
├── app/
│   ├── __init__.py          # Required for imports
│   ├── data/
│   │   └── config.json      # Static files preserved
│   └── functions/
│       ├── __init__.py      # Required for imports
│       └── handler.py
└── pyproject.toml
```

SST automatically preserves your directory structure and includes static files, ensuring both module imports and file access work correctly in Lambda.

## Advanced

### File Inclusion and Exclusion

SST uses a smart two-layer system to determine which files are included in your Lambda deployment package, ensuring optimal package size while including everything your function needs.

### Custom Configuration (pyproject.toml)

You can explicitly control file inclusion/exclusion by adding a `[tool.sst]` section to your `pyproject.toml`:

```toml title="pyproject.toml"
[project]
name = "my-lambda-project"
dependencies = ["requests>=2.31.0"]

[tool.sst]
# Files to explicitly include (overrides standard exclusions)
include = [
    "config/*.json",           # Include all JSON config files
    "templates/*.html",        # Include HTML templates
    "static/critical.css",     # Include specific critical CSS
    ".env.production",         # Include production environment file
    "data/**/*.csv"            # Include all CSV files in data subdirectories
]

# Files to explicitly exclude (in addition to standard exclusions)
exclude = [
    "tests/**",               # Exclude all test files
    "docs/**",                # Exclude documentation
    "*.log",                  # Exclude log files
    "temp/**",                # Exclude temporary files
    "scripts/dev-*",          # Exclude development scripts
    "*.backup",               # Exclude backup files
    "large-datasets/**"       # Exclude large data files
]
```

:::note
Include patterns take precedence over exclude patterns. If a file matches both an include and exclude pattern, it will be included.
:::

### Standard Exclusions (Automatic)

When no custom configuration is found, SST applies sensible defaults:

**Automatically Included:**
- All Python files (`.py`)
- All directories containing Python files
- Package structure files (`__init__.py`)
- Static files in common directories (`data/`, `config/`, `templates/`, `static/`, `assets/`)
- Essential configuration files (`requirements.txt`)

**Automatically Excluded:**
- **Build artifacts**: `__pycache__/`, `*.pyc`, `*.pyo`, `*.pyd`, `*.egg-info/` (from both source code and installed dependencies)
- **Version control**: `.git/`, `.gitignore`, `.gitattributes`
- **Virtual environments**: `.venv/`, `venv/`, `env/`
- **IDE files**: `.vscode/`, `.idea/`, `*.swp`, `.DS_Store`
- **Test artifacts**: `.pytest_cache/`, `.coverage`, `htmlcov/`
- **Documentation**: `README.md`, `CHANGELOG.md`, `LICENSE`
- **Development config**: `pyproject.toml`, `setup.py`, `tox.ini`, `Makefile`
- **Development dependencies**: `requirements-dev.txt`, `.pre-commit-config.yaml`
- **Node.js artifacts**: `node_modules/`, `package-lock.json` (for mixed projects)
- **Temporary files**: `*.log`, `*.tmp`, `tmp/`, `temp/`
- **SST cache**: `.sst/`

### Examples

**Include config files, exclude large data:**
```toml title="pyproject.toml"
[tool.sst]
include = ["config/*.json", "data/**/*.csv"]
exclude = ["datasets/**", "notebooks/**", "*.pkl"]
```

## Troubleshooting

### Common Issues

**Handler Not Found**
```
Error: Handler 'src/myapp/handler.py' not found
```

**Solution**: Verify the handler path in your Function configuration matches your file structure:

```typescript
// File: src/myapp/handler.py, function: main
new Function("MyFunction", {
  handler: "src/myapp/handler.main"  // Correct path
})
```

**Import Errors**
```
ModuleNotFoundError: No module named 'mymodule'
```

**Solutions**:
1. Ensure `__init__.py` files exist in all package directories
2. Use absolute imports instead of relative imports
3. Verify the module is in the correct directory structure

**File Not Found**
```
FileNotFoundError: [Errno 2] No such file or directory: 'config.json'
```

**Solutions**:
1. Use paths relative to `__file__` instead of working directory
2. Verify the file exists in your project structure
3. Check that the file is not in an excluded directory
4. Add the file to the include list in `pyproject.toml` if needed:
   ```toml
   [tool.sst]
   include = ["path/to/your/file.json"]
   ```

**Package Too Large**
```
Unzipped size must be smaller than 262144000 bytes
```

**Solutions**:
1. Exclude unnecessary files:
   ```toml
   [tool.sst]
   exclude = [
       "large-data/**",
       "*.pkl",
       "datasets/**",
       "notebooks/**"
   ]
   ```
2. Use container mode for large dependencies
3. Check what files are being included by reviewing the build output in `.sst/artifacts/`

**Required File Missing from Package**
```
FileNotFoundError: config.json not found in Lambda
```

**Solutions**:
1. Explicitly include the file:
   ```toml
   [tool.sst]
   include = ["config.json", "data/**/*.json"]
   ```
2. Move the file to a standard location (`data/`, `config/`, `static/`)
3. Verify the file path is correct relative to your handler

:::note
On macOS, if deploys hang after the first successful deploy, clean the artifact cache: `rm -rf .sst/artifacts/*-src && sst deploy`. This is caused by a macOS/APFS filesystem quirk with hardlinked directories.
:::

### Container Mode

For large dependencies (numpy, pandas, ML libraries), use container mode:

```typescript
new Function("MLFunction", {
  handler: "src/ml/handler.predict",
  runtime: "python3.11",
  container: {
    dockerfile: "Dockerfile"
  }
})
```

This bypasses Lambda's 250MB zip limit and provides more flexibility for complex dependencies.

:::note
When migrating from previous versions, existing projects work without changes. Add `[tool.sst]` configuration only when you need to customize file inclusion.
:::
